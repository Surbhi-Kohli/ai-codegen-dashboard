AI Dashboard Design - Metrics & Formulas

Metrics, Formulas & Sources

Complete breakdown of every metric, how it's calculated, and where the data comes from

Pillar 1: Delivery Speed & Reliability

Cycle Time (per issue) Jira + GitHub
Cycle Time = PR Merged Timestamp − Issue "In Progress" Timestamp

Broken into sub-stages:
  • Coding Time = First PR Created − Issue moved to "In Progress" | Jira transitions + GitHub PR
  • Review Time = PR Approved Timestamp − First Review Requested Timestamp | GitHub PR events
  • Waiting Time = Cycle Time − Coding Time − Review Time | Derived (includes queue time, CI, merge delays)
Velocity (per sprint) Jira
Velocity = Σ Story Points of issues completed in sprint

Trend: compare sprint-over-sprint. Segment by AI-assisted vs non-AI issues.
Sprint Completion Rate Jira
Completion Rate = (Issues Done in Sprint / Issues Committed to Sprint) × 100%
PR Merge Rate GitHub
Merge Rate = (PRs Merged / PRs Opened) × 100%

Per time window. Low rate may signal quality issues or scope churn.
PR Throughput GitHub
Throughput = Count of PRs merged per time window

Pillar 2: Where Time Is Spent & Where Work Gets Stuck

Time-in-Stage Distribution Jira + GitHub
% in Stage = (Avg Time in Stage / Avg Total Cycle Time) × 100%

Computed at three levels:
  • Team = Average across all issues in time window
  • Developer = Average across issues assigned to developer
  • Issue = Actual time per stage for a single issue
Review Queue Depth GitHub
Queue Depth = Count of open PRs with review requested but no review submitted

Point-in-time snapshot. High depth = bottleneck.
Review Pickup Time GitHub + Webex
Pickup Time = First Review Comment Timestamp − Review Requested Timestamp

Supplemented by Webex:
Webex Response Time = First Reply in Thread − Review Request Message Timestamp

Webex response time captures acknowledgment even before formal GitHub review begins.
Rework Rate GitHub
Rework Rate = (PRs with "Changes Requested" / Total PRs Reviewed) × 100%

Review Rounds = Count of review iterations before approval
Blocked Time (Jira) Jira
Blocked Time = Time issue spent in "Blocked" status or with blocker flag

Pillar 3: AI Impact

AI Code Contribution git-ai
AI % per Commit = (AI-attributed lines / Total lines changed) × 100%

AI % per PR = Σ AI lines across commits / Σ Total lines across commits × 100%

AI % Team (period) = Σ AI lines merged / Σ Total lines merged × 100%

Segmentable by: developer, repo, agent (Cursor/Claude/Copilot), model.
AI vs Human: Review Comment Density git-ai + GitHub
AI Comment Density = Review comments on AI-attributed lines / AI lines in PR

Human Comment Density = Review comments on human lines / Human lines in PR

Compared within the same PR. Requires matching GitHub review comment line positions against git-ai attribution ranges.
AI vs Human: Revision Rounds git-ai + GitHub
AI Revision Rate = Commits that modify AI-attributed lines after review / Total review rounds

Human Revision Rate = Commits that modify human lines after review / Total review rounds

Measures whether AI code requires more post-review fixes than human code.
AI vs Human: Review Turnaround git-ai + GitHub
Avg Review Time (high-AI PRs) vs Avg Review Time (low-AI PRs)

Threshold-based: PRs with >50% AI code vs <20% AI code. Shows if reviewers treat AI-heavy PRs differently.
AI Tool & Model Usage git-ai
Usage by Agent = Count of commits per agent (Cursor, Claude Code, Copilot, etc.)

Usage by Model = Count of commits per model (Sonnet, Opus, GPT-4, etc.)

Track adoption trends and which tools the team gravitates toward.
AI Productivity Impact git-ai + Jira
Avg Coding Time (AI-assisted issues) vs Avg Coding Time (non-AI issues)

Avg Cycle Time (AI-assisted issues) vs Avg Cycle Time (non-AI issues)

An issue is "AI-assisted" if any of its linked PRs contain git-ai attributed code. Compare same issue types (bug vs bug, story vs story) for fairness.

Pillar 4: AI Quality & Oversight

AI Code Defect Rate git-ai + GitHub
Defect Rate = (Bug-fix PRs modifying AI-attributed lines / Total PRs with AI code) × 100%

Tracks whether AI-generated code produces bugs at a higher rate than human code. Bug-fix PRs identified via Jira issue type (Bug) linked to the PR.
AI Code Revert Rate git-ai + GitHub
Revert Rate = (Reverted commits with AI-attributed lines / Total commits with AI code) × 100%

High revert rate suggests AI output is being accepted without sufficient review.
Unmodified AI Code Ratio (Copy-Paste Indicator) git-ai
Unmodified Ratio = (AI lines unchanged from generation to merge / Total AI lines) × 100%

Compares git-ai checkpoint snapshot against final committed code. Interpretation requires context: A high ratio may indicate blind acceptance (risky) OR excellent spec/prompt engineering where the AI output was correct first try (healthy). Never flag in isolation — evaluate alongside test coverage, review engagement, and defect history.
Review Depth on AI Code git-ai + GitHub
Review Comments per AI Line vs Review Comments per Human Line (within same PR)

Avg Review Comment Length, trended over time, segmented by AI-heavy (>50%) vs non-AI (<20%) PRs

Declining review depth on AI-heavy PRs is a leading indicator that reviewers are rubber-stamping AI output.
AI Review Blind Acceptance Rate GitHub
Blind Accept Rate = (AI review threads resolved in <2 min or with no reply / Total AI review threads) × 100%

Tracks whether developers are dismissing automated AI review suggestions (e.g., chatgpt-codex-connector) without reading them.
Prompt-to-Fix Cycle git-ai + GitHub
Follow-up Fixes = Count of commits within 24h that modify AI-attributed lines from the same PR

Frequent immediate fixes suggest the original AI output was merged prematurely.
Test Coverage on AI Code git-ai + GitHub
Test Ratio (AI PRs) = Test lines added / AI code lines added

Test Ratio (non-AI PRs) = Test lines added / Code lines added

Ensures AI-generated code is tested with the same rigor as human-written code. Test files identified by path heuristic (test/, utest/, itest/).
Knowledge Concentration Risk git-ai + git blame
Concentration = % of files in module where >80% of recent changes come from single author + AI

Flags bus-factor risk where one developer + AI is writing most of a module, reducing shared understanding across the team.

Webex Communication Metrics

Review Request Response Time Webex
Response Time = First Thread Reply Timestamp − Original Message Timestamp

Tracked in the team's dedicated review Webex space. Measures how quickly the team acknowledges review requests in their communication channel.
Communication-to-Action Gap Webex + GitHub
Gap = First GitHub Review Activity − First Webex Thread Reply on same PR

Measures lag between "I'll look at it" (Webex ack) and actually reviewing (GitHub activity). Large gap = context switching or deprioritization.
